## 🧠 Why This Is The Right Approach

### The Problem with Everything We've Built So Far:
Our V2.2 tool reads HAR files. HAR files only contain what a human *already did*. If the human never clicked the "Admin Settings" tab, or never filled out the "Custom Object Form", those endpoints **don't exist in the HAR**. Our tool is blind to 60-70% of the attack surface.

### The Solution: The 4-Phase AI Agent

```
Phase 1: EXPLORE          Phase 2: UNDERSTAND        Phase 3: PLAN           Phase 4: TEST
┌─────────────┐      ┌─────────────────────┐    ┌────────────────┐    ┌──────────────┐
│ Playwright   │─────▶│ Vision LLM builds    │───▶│ AI generates   │───▶│ Execute safe │
│ clicks every │      │ a Feature Inventory: │    │ targeted tests │    │ payloads per │
│ link, tab,   │      │ pages, forms, roles, │    │ based on what  │    │ feature      │
│ button       │      │ inputs, permissions  │    │ it understands │    │ discovered   │
└─────────────┘      └─────────────────────┘    └────────────────┘    └──────────────┘
```

## 📐 The Architecture

### Phase 1: Autonomous Explorer (`src/autonomous_explorer.py`)
- Logs into the portal using credentials from `credentials.yaml`
- Systematically clicks every navigation link, tab, button, and menu item
- At each page, captures: Screenshot, DOM structure, all visible input fields, all clickable elements
- Sends to Vision LLM: *"What is this page? What can a user do here?"*
- Builds a **Site Map** (graph of all pages and their relationships)

### Phase 2: Feature Inventory Builder (`src/feature_inventory.py`)
- Aggregates all LLM page analyses into a structured inventory:
  - Every page and its purpose
  - Every form and its fields (with field types)
  - Every table/grid and its columns
  - Every role difference (Admin sees X, User sees Y)
- This becomes the **attack surface model**

### Phase 3: Smart Test Planner (`src/test_planner.py`)
- Reads the Feature Inventory and maps features to risks:
  - Search box → SOQL Injection risk
  - File upload → Path traversal / malware risk
  - Record ID in URL → BOLA/IDOR risk
  - Rich text field → Stored XSS risk
  - "Delete" button without confirmation → Business logic flaw risk
- Generates a prioritized test plan

### Phase 4: Safe Payload Executor (`src/safe_executor.py`)
- Uses **non-destructive, clearly fake payloads** for exploration:
  - Instead of `<script>alert(1)</script>` → uses `SF_TEST_XSS_PROBE_7f3a`
  - Instead of `' OR 1=1--` → uses `SF_TEST_SQLI_PROBE_2b8c`
- Checks visually if the probe string reflects anywhere on the page
- Only escalates to real payloads if a probe reflects
